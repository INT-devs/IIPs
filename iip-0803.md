# IIP-803: Lightning Onion Routing

## Abstract

This IIP specifies the onion routing protocol for INTcoin Lightning
payments, adapting BOLT #4 (Sphinx) with ML-KEM-1024 (Kyber1024)
key encapsulation replacing ECDH for post-quantum security at each
hop.

## Motivation

Lightning payments are routed through multiple hops. Each hop should
only know its predecessor and successor, not the full route. The
Sphinx onion routing protocol provides this property using layered
encryption. INTcoin replaces the X25519/ECDH key exchange at each
hop with ML-KEM-1024 to maintain post-quantum security.

## Specification

### Onion Packet Structure

```
+---------+--------------------------------------------+------+
| Version | Encrypted routing info (per-hop payloads)  | HMAC |
| 1 byte  | variable                                   | 32 B |
+---------+--------------------------------------------+------+
```

### Per-Hop Key Exchange

At each hop, the sender:
1. Generates an ephemeral ML-KEM-1024 keypair
2. Encapsulates a shared secret to the hop's public key
3. Derives encryption keys from the shared secret via HKDF-SHA3-256
4. Encrypts the per-hop payload with ChaCha20-Poly1305

The hop node:
1. Decapsulates the shared secret using its ML-KEM secret key
2. Derives the same encryption keys
3. Decrypts its per-hop payload
4. Peels the onion layer and forwards to the next hop

### Per-Hop Payload

| Field | Size | Description |
|-------|------|-------------|
| `channel_id` | 8 bytes | Short channel ID for next hop |
| `amount_to_forward` | 8 bytes | Amount in base units |
| `outgoing_cltv_value` | 4 bytes | CLTV expiry for next hop |
| `padding` | variable | Zero-padded to fixed size |

### Size Impact

| Component | Bitcoin | INTcoin |
|-----------|---------|---------|
| Ephemeral pubkey | 33 B | 1,568 B (ML-KEM ciphertext) |
| Per-hop overhead | ~65 B | ~1,600 B |
| Max hops (20) | ~1,300 B | ~32,000 B |

The larger onion packet size is acceptable for Lightning where
packets are transmitted over direct peer connections (not broadcast).

### Blinded Routes

For receiver privacy, the final hops of a route can be blinded.
The sender constructs the onion without knowing the final
destination's identity — only a blinded route hint provided in
the invoice or offer.

Blinding uses ML-KEM-1024 for the blinding factor exchange,
maintaining post-quantum security for the receiver's identity.

### Error Reporting

Intermediate nodes report routing failures via encrypted error
messages sent backwards along the route. Each hop decrypts one
layer, using the same shared secret derived during forward
routing.

### Path Finding

The router selects paths based on:
- Channel capacity and liquidity
- Fee rates (base fee + proportional fee)
- CLTV delta requirements
- Historical success/failure rates (mission control)

Multi-path payments split large amounts across multiple routes
for better success probability and privacy.

## Security Considerations

- ML-KEM-1024 provides IND-CCA2 security (NIST Level 5)
- Each hop sees only predecessor/successor, not full route
- Payment hash correlation: intermediaries can correlate
  same-preimage payments (addressed by point-time-lock
  contracts in future IIP)
