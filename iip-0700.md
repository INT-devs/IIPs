# IIP-700: Institutional Custody Module

## Abstract

This IIP specifies the optional institutional custody module for
INTcoin, providing Role-Based Access Control (RBAC), tamper-evident
audit logging, spending policies with multi-party approval,
address whitelisting with AML screening, and external signer
integration. The module is opt-in at build time
(`-DBUILD_INSTITUTIONAL=ON`) and is not included in default builds.

## Motivation

Institutional adoption of cryptocurrency requires compliance
infrastructure: audit trails, separation of duties, spending
controls, and integration with existing governance workflows.
Building these directly into the node software (rather than as
external middleware) provides tighter integration, lower latency,
and a unified security boundary.

## Specification

### Build flag

The institutional module compiles only when the CMake flag
`BUILD_INSTITUTIONAL=ON` is set. Default builds (`OFF`) do not
include any institutional code — calling institutional RPCs returns
`Method not found`.

### RBAC (Role-Based Access Control)

#### Users

| Field | Description |
|-------|-------------|
| userid | Unique string identifier |
| name | Display name |
| email | Contact email |
| password_hash | Argon2id hash of password |
| roles | List of assigned role names |
| mfa_enabled | TOTP MFA enrollment status |
| failed_logins | Counter for lockout |
| daily_spending | Running total for policy evaluation |

#### Roles

Roles define permissions. Built-in roles:

| Role | Permissions |
|------|------------|
| admin | All operations |
| operator | Node management, wallet operations |
| viewer | Read-only access to all data |
| signer | Transaction signing only |
| auditor | Audit log access only |

Custom roles may be created.

#### MFA

TOTP MFA follows RFC 6238:

- HMAC-SHA1, 6 digits, 30-second window, +/- 1 step skew tolerance.
- `rbac_enrollmfa` returns base32 secret + otpauth:// URI.
- `rbac_confirmmfa` validates the first code before enabling.
- Constant-time comparison prevents timing attacks.

### Audit logging

Every significant operation generates a tamper-evident audit event.

#### Chain structure

```
event.hash = SHA3-256(event.prev_hash || canonical_json(event))
```

The chain tip is persisted to disk on every event. `verifyauditlog`
walks the chain and verifies every event's hash against its
predecessor.

#### Event fields

| Field | Description |
|-------|-------------|
| id | Monotonic event ID |
| timestamp | ISO 8601 |
| category | RBAC, WALLET, POLICY, WHITELIST, SIGNER, ... |
| action | The operation performed |
| actor | User ID or "system" |
| details | Structured JSON of the operation |
| prev_hash | SHA3-256 hash of prior event |
| hash | SHA3-256(prev_hash \|\| canonical_json) |

#### Storage

- `<datadir>/institutional/audit/audit-YYYY-MM-DD.jsonl` per day.
- `audit_chain_tip.txt` persists the latest hash across restarts.

### Spending policies

Policies define rules evaluated before any outbound transaction.

| Policy field | Description |
|-------------|-------------|
| max_single | Maximum single transaction amount |
| max_daily | Maximum daily aggregate per user/wallet |
| require_approval | Number of approvals needed |
| approvers | List of user IDs who may approve |
| whitelist_only | Restrict destinations to whitelist |

Evaluation: `policy_evaluate <amount> <destination>` runs all
applicable policies and returns allow/deny with reasons.

### Address whitelist

| Field | Description |
|-------|-------------|
| address | INTcoin address (IIP-0400 format) |
| label | Description |
| added_by | User who whitelisted |
| wallet_id | Scope (optional) |

Adding an address consults the AML screener before queueing. If the
screener denies, the address is blocked with an audit event
(`WHITELIST_BLOCKED`).

### AML screening

Pluggable interface:

```cpp
class IAMLScreener {
    virtual AMLResult Screen(const std::string& address) = 0;
};
```

Built-in implementations:
- **BlocklistAMLScreener** — in-memory deny list.
- **No screener** — allow all (default when none installed).

Errors during screening are treated as denials (fail-closed).

### External signers (HSMSigner)

Software-backed key-isolated signer storing Dilithium5 keypairs in
an AES/ChaCha20-Poly1305 encrypted file. PIN derived via
HKDF-SHA3-256.

| File | Description |
|------|-------------|
| `<datadir>/signers/<name>.signer` | Encrypted keypair store |

Signs with real 4,627-byte ML-DSA-87 signatures. Emits an audit
event on every connect, unlock, and sign operation.

Note: this is a software backend. Real PKCS#11 HSM support is
blocked on HSM vendors shipping `CKM_ML_DSA` firmware.

### State files

All under `<datadir>/institutional/`:

| File | Contents |
|------|----------|
| `rbac.json` | Users, roles, sessions, password hashes |
| `policies.json` | Spending policies, pending approvals |
| `whitelist.json` | Whitelisted addresses, daily counters |
| `audit/audit-YYYY-MM-DD.jsonl` | Daily audit log |
| `audit/audit_chain_tip.txt` | Current chain tip hash |
| `signers/<name>.signer` | Encrypted signer keystores |

All non-audit state files use atomic write (write `.tmp` + fsync +
rename).

### RPC interface

| Category | Commands |
|----------|----------|
| Audit | `getauditlog`, `getauditstats`, `verifyauditlog`, `getauditexport` |
| RBAC | `rbac_createuser`, `rbac_listusers`, `rbac_listroles`, `rbac_assignrole`, `rbac_enrollmfa`, `rbac_confirmmfa`, `rbac_disablemfa` |
| Policy | `policy_list`, `policy_evaluate`, `policy_pendingapprovals`, `policy_approve` |
| Whitelist | `whitelist_add`, `whitelist_list`, `whitelist_check` |
| AML | `amlcheck` |
| Dashboard | `compliancedashboard` |
| Signers | `signer_list`, `signer_enumeratekeys` |

## Rationale

Building custody features into the node (rather than external
middleware) eliminates the trust boundary between the signing
environment and the policy engine. The opt-in build flag keeps the
default binary lean for non-institutional users.

The tamper-evident audit chain uses the same SHA3-256 hash function
as the rest of the protocol, and the chain-tip persistence ensures
continuity across restarts.

## Backwards Compatibility

The module is opt-in and does not affect consensus, P2P, or wallet
behaviour for non-institutional builds.
