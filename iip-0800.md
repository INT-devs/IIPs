# IIP-800: Lightning Network Protocol

## Abstract

This IIP defines the Lightning Network protocol for INTcoin, enabling
off-chain payment channels with sub-second settlement. The protocol
adapts the BOLT (Basis of Lightning Technology) specification for
post-quantum cryptography, replacing ECDSA with ML-DSA-87 (Dilithium5)
for all channel signatures and ML-KEM-1024 (Kyber1024) for key exchange
in onion routing.

## Motivation

INTcoin's 2-minute block time handles most on-chain use cases, but
Lightning adds:

1. **Sub-second payments** for point-of-sale and streaming use cases
2. **Micropayments** below the on-chain dust threshold
3. **Scalability** — thousands of payments per channel with only two
   on-chain transactions (open and close)
4. **Privacy** — onion-routed payments reveal less to intermediaries
   than on-chain transactions

Lightning is developed as a **separate repository** and connects to
INTcoin Core via RPC or the IPC interface.

## Specification Overview

The INTcoin Lightning Network follows the BOLT specification with
these post-quantum modifications:

| BOLT | INTcoin Adaptation | IIP |
|------|-------------------|-----|
| BOLT #2 | Channel state machine with Dilithium5 signatures | IIP-0801 |
| BOLT #3 | Commitment transactions with ML-DSA-87 | IIP-0801 |
| BOLT #4 | Onion routing with Kyber1024 packets | IIP-0803 |
| BOLT #7 | Gossip protocol for channel discovery | IIP-0804 |
| BOLT #9 | Feature bits for post-quantum capabilities | This IIP |
| BOLT #11 | Invoice protocol with int/tint prefixes | IIP-0802 |
| BOLT #12 | Offers protocol | IIP-0805 |

### Post-Quantum Cryptographic Substitutions

| Bitcoin Lightning | INTcoin Lightning | Size Impact |
|-------------------|-------------------|-------------|
| ECDSA signatures (72 bytes) | ML-DSA-87 signatures (4627 bytes) | ~64x larger |
| secp256k1 pubkeys (33 bytes) | ML-DSA-87 pubkeys (2592 bytes) | ~79x larger |
| X25519 key exchange (32 bytes) | ML-KEM-1024 encapsulation (1568 bytes) | ~49x larger |
| ECDH shared secrets (32 bytes) | ML-KEM shared secrets (32 bytes) | Same |

### Feature Bits

| Bit | Name | Description |
|-----|------|-------------|
| 0/1 | `option_data_loss_protect` | Channel reestablishment data loss protection |
| 4/5 | `option_upfront_shutdown_script` | Require shutdown script at open |
| 8/9 | `option_dilithium_sigs` | **Required** — ML-DSA-87 channel signatures |
| 12/13 | `option_static_remotekey` | Static remote key for simplified backup |
| 14/15 | `option_kyber_onion` | **Required** — Kyber1024 onion routing |
| 24/25 | `option_anchors` | Anchor outputs for fee bumping |

### Chain Monitor

The Lightning node monitors the blockchain for:
- Channel close transactions (cooperative and force)
- HTLC timeout/success transactions
- Breach attempts (revoked commitment broadcast)
- Deadline tracking for time-sensitive transactions

Automatic sweep broadcasting and fee bumping ensure funds are
recovered before timelocks expire.

## Compatibility

INTcoin Lightning is **not compatible** with Bitcoin Lightning due
to different signature schemes, address formats, and onion routing
cryptography. Cross-chain atomic swaps between INTcoin Lightning
and Bitcoin Lightning require a bridge protocol (future IIP).
