# IIP-404: Partially Signed Transactions (PST)

## Abstract

This IIP specifies the Partially Signed Transaction (PST) format for
INTcoin, adapted from BIP-174 (PSBT) for ML-DSA-87 post-quantum
signatures. PST enables multi-party signing workflows, hardware
wallet integration, and offline signing without requiring any party
to hold all private keys simultaneously.

## Motivation

Secure transaction construction often requires separating the roles
of creation, funding, signing, and broadcasting. A standardised
intermediate format allows these roles to be performed by different
software or devices. BIP-174's PSBT model is adapted for INTcoin's
larger key and signature sizes.

## Specification

### Magic bytes

| Format | Magic |
|--------|-------|
| PST binary | `pst\xff\x00` (5 bytes) |
| PST Base64 prefix | `cHN0` |

**Divergence from BIP-174:** BIP-174 uses `psbt\xff` (5 bytes).
INTcoin uses `pst\xff\x00` to prevent cross-chain confusion.

### Structure

A PST is a key-value map structure containing:

1. **Global map** — unsigned transaction, version info.
2. **Per-input maps** — UTXO data, key derivation paths, partial
   signatures, sighash types.
3. **Per-output maps** — key derivation paths for change detection.

### Key types

| Key | Scope | Description |
|-----|-------|-------------|
| 0x00 | Global | Unsigned transaction |
| 0x01 | Input | Non-witness UTXO |
| 0x02 | Input | Witness UTXO |
| 0x03 | Input | Partial signature (ML-DSA-87) |
| 0x04 | Input | Sighash type |
| 0x06 | Input | HD key derivation path |
| 0x02 | Output | HD key derivation path |

### Partial signatures

Partial signatures use the ML-DSA-87 scheme (IIP-0300):

| Component | Size |
|-----------|------|
| Signing public key (map key) | 2,592 bytes |
| Signature (map value) | 4,627 bytes |

Multiple signers each add their own partial signature keyed by their
public key.

### Workflow roles

| Role | Responsibility |
|------|---------------|
| **Creator** | Constructs PST with inputs and outputs |
| **Updater** | Adds UTXO info, key paths, sighash types |
| **Signer** | Adds ML-DSA-87 partial signatures |
| **Combiner** | Merges PSTs from multiple signers |
| **Finalizer** | Constructs final scriptSig from signatures |
| **Extractor** | Extracts the fully-signed transaction for broadcast |

### Serialization

PSTs are serialized as key-value maps separated by `0x00` bytes:

```
<magic> <global_map> 0x00 <input_map_0> 0x00 ... <output_map_0> 0x00 ...
```

Each key-value pair is: `<key_len> <key> <value_len> <value>`.
Lengths use CompactSize encoding.

### Text encoding

For text transport (copy-paste, QR codes), PSTs are Base64-encoded.
The Base64 string always starts with `cHN0` (the encoding of
`pst\xff\x00`).

Hex encoding is also supported for RPC interfaces.

### Analysis

The `analyzepst` RPC returns signing status:

```json
{
  "inputs_total": 2,
  "inputs_signed": 1,
  "inputs_finalized": 0,
  "is_complete": false,
  "is_signable": true,
  "missing_sigs": ["int1q..."],
  "fee": 100000
}
```

### RPC interface

| Command | Description |
|---------|-------------|
| `createpst` | Create unsigned PST |
| `decodepst` | Decode PST to JSON |
| `analyzepst` | Analyse signing status |
| `combinepst` | Combine multiple PSTs |
| `finalizepst` | Finalize signed PST |
| `walletcreatedfundedpst` | Create funded PST from wallet |
| `walletprocesspst` | Sign PST with wallet keys |
| `sendpst` | Finalize and broadcast |

### Current limitations

- `combinepst` and `finalizepst` are under active development.
  Multi-signer workflows are not yet fully operational.
- BIP-32 derivation paths in PST are not supported (Dilithium5
  does not support standard BIP-32 derivation — see IIP-0301).

## Divergences from BIP-174

| Aspect | BIP-174 (PSBT) | IIP-0404 (PST) |
|--------|---------------|----------------|
| Magic bytes | `psbt\xff` | `pst\xff\x00` |
| Signature size | ~72 bytes (DER ECDSA) | 4,627 bytes (ML-DSA-87) |
| Public key size | 33 bytes (compressed) | 2,592 bytes |
| Key derivation | BIP-32 paths | IIP-0301 paths (SHA3-256) |
| Multisig | OP_CHECKMULTISIG | Not supported; per-signer independent signatures |

## Rationale

The PSBT model is preserved because the role-based workflow (Creator,
Updater, Signer, Combiner, Finalizer, Extractor) is independent of
the signature algorithm. Only the signature and key sizes change. The
distinct magic bytes prevent a PST from being misinterpreted as a
PSBT by Bitcoin software.

## Backwards Compatibility

Not applicable. PST is a new format with no predecessor in INTcoin.
