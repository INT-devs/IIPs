# IIP-300: ML-DSA-87 Digital Signature Scheme

## Abstract

This IIP specifies the use of ML-DSA-87 (formerly Dilithium5) as the
sole digital signature algorithm in INTcoin for transaction signing,
block signing, and message authentication. ML-DSA-87 is a
NIST-standardised (FIPS 204) lattice-based signature scheme providing
security level 5 (256-bit classical, post-quantum resistant).

## Motivation

INTcoin requires a signature scheme that is secure against both
classical and quantum computer attacks. ML-DSA-87 was selected as the
highest security level of the NIST post-quantum digital signature
standard, providing long-term security as quantum computing matures.

Unlike ECDSA, ML-DSA-87 cannot derive public keys from secret keys.
This fundamental property shapes the key storage model — both keys
must always be stored together.

## Specification

### Algorithm identification

| Field | Value |
|-------|-------|
| NIST standard | FIPS 204 |
| Algorithm name | ML-DSA-87 |
| Legacy name | CRYSTALS-Dilithium (Level 5) |
| liboqs identifier | `OQS_SIG_alg_ml_dsa_87` |
| Security level | NIST Level 5 (256-bit classical) |

### Key and signature sizes

| Component | Size (bytes) |
|-----------|-------------|
| Public key (CPubKey) | 2,592 |
| Secret key (CKey) | 4,896 |
| Signature | 4,627 |

### Key generation

Key pairs are generated via liboqs:

```
OQS_SIG *sig = OQS_SIG_new("ML-DSA-87")
OQS_SIG_keypair(sig, public_key, secret_key)
```

For HD wallet derivation, deterministic key generation is used:

```
GenerateKeyPairFromSeed(seed_32bytes, secret_key, public_key)
```

Deterministic generation uses a thread-local SHAKE256 XOF instance
seeded with the 32-byte input. The SHAKE256 output stream replaces
the RNG source for `OQS_SIG_keypair`, producing identical keys for
identical seeds.

### Signing

```
OQS_SIG_sign(sig, signature, &sig_len,
             message, message_len, secret_key)
```

The message is the 32-byte SHA3-256 sighash digest computed per
IIP-0100. The output signature is always exactly 4,627 bytes.

### Verification

```
OQS_SIG_verify(sig, message, message_len,
               signature, sig_len, public_key)
```

Returns `OQS_SUCCESS` if and only if the signature is valid for the
given message and public key.

### Key storage model

Because ML-DSA-87 cannot derive the public key from the secret key,
`CKey` stores both:

```cpp
class CKey {
    std::vector<uint8_t> m_secretKey;  // 4,896 bytes
    CPubKey m_pubKey;                  // 2,592 bytes
};
```

`CKey::Set()` requires both keys. `CKey::GetPubKey()` returns the
stored public key directly (no derivation).

### Hash functions used

| Purpose | Algorithm |
|---------|-----------|
| Sighash computation | SHA3-256 |
| Address derivation (pubkey hash) | RIPEMD160(SHA3-256(pubkey)) |
| Deterministic key entropy expansion | SHAKE256 |
| HD key derivation chain | SHA3-256 (see IIP-0301) |

### Error handling

| Error | Condition |
|-------|-----------|
| INVALID_KEY_SIZE | Public key != 2,592 bytes or secret key != 4,896 bytes |
| INVALID_SIGNATURE_SIZE | Signature != 4,627 bytes |
| VERIFICATION_FAILED | ML-DSA-87 verify returned failure |
| SIGNING_FAILED | ML-DSA-87 sign returned failure |

### Initialization

`crypto::Initialize()` must be called before any cryptographic
operations. This calls `OQS_init()` and creates the global signature
scheme instance. `crypto::Shutdown()` releases resources.

## Rationale

ML-DSA-87 (level 5) was chosen over level 2 or level 3 variants for
maximum long-term security. The larger key and signature sizes (2,592
+ 4,627 bytes per signed input vs ~33 + ~72 bytes for ECDSA) increase
transaction size substantially but are an accepted trade-off for a
chain designed with post-quantum security as its primary goal.

SHAKE256 is used for deterministic key generation rather than
HMAC-SHA512 (as in BIP-32) because it is part of the same SHA-3
family and provides a natural XOF (extendable output function) for
entropy expansion.

## Backwards Compatibility

Not applicable. ML-DSA-87 has been the only signature scheme since
genesis.
