# IIP-503: Multiprocess IPC Architecture

## Abstract

This IIP specifies the inter-process communication (IPC) architecture
for splitting intcoind into separate processes: node, wallet, and
miner. Process isolation improves security, reliability, and
operational flexibility.

## Motivation

A single-process architecture means a wallet bug can crash the node,
a miner hang can block RPC, and restarting the wallet requires
restarting the entire node. Process isolation addresses all three:

- **Security:** Wallet keys are isolated from network-facing code
- **Reliability:** One process crashing doesn't affect others
- **Flexibility:** Wallet can connect to a remote node

## Specification

### Process Split

| Process | Responsibilities |
|---------|-----------------|
| `intcoind-node` | P2P networking, block validation, mempool, chainstate |
| `intcoind-wallet` | Key management, coin selection, transaction building |
| `intcoind-miner` | Block template, PoW computation (already separate) |

### IPC Interface

Communication via Unix domain sockets (local) or TCP (remote):

```
intcoind-wallet ←→ intcoind-node (wallet operations)
intcoind-miner  ←→ intcoind-node (template + submission)
intcoin-qt      ←→ intcoind-node (GUI mode)
```

### Serialisation

Messages serialised using a compact binary format (not JSON-RPC)
for performance. Each message has:
- 4-byte message type
- 4-byte payload length
- Variable payload (INTcoin serialisation format)

### Key Operations

| Operation | Direction | Description |
|-----------|-----------|-------------|
| `GetBalance` | Wallet → Node | Query UTXO set for wallet addresses |
| `SendTransaction` | Wallet → Node | Submit signed transaction to mempool |
| `GetBlockTemplate` | Miner → Node | Request block template |
| `SubmitBlock` | Miner → Node | Submit mined block |
| `NotifyBlock` | Node → Wallet | New block notification |
| `NotifyTx` | Node → Wallet | Relevant transaction notification |

### Backward Compatibility

Single-process mode remains the default. The `--multiprocess` flag
enables process separation. Existing RPC interface unchanged.

## Reference Implementation

- IPC interface: `src/ipc/` (planned)
