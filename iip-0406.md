# IIP-406: Payjoin

## Abstract

This IIP will specify the Payjoin protocol for INTcoin, adapted from
BIP-78. Payjoin (also known as Pay-to-Endpoint or P2EP) is a
privacy-enhancing transaction construction protocol where both sender
and receiver contribute inputs to a payment transaction, breaking the
common-input-ownership heuristic used by chain analysis.

## Motivation

Standard transactions reveal sender and receiver through input/output
analysis. The common-input-ownership heuristic assumes all inputs in
a transaction belong to the same entity. Payjoin breaks this
assumption by having the receiver add their own inputs to the
transaction, making it indistinguishable from a regular multi-input
spend.

## Status

**Draft — not yet implemented.** This IIP is a placeholder for the
privacy roadmap. The specification will be completed when development
begins.

## Specification (preliminary)

The BIP-78 Payjoin model will be adapted with the following
anticipated divergences:

- **Transaction format:** BIP-78 uses PST (PSBT in Bitcoin) as the
  intermediate format for the multi-party construction. INTcoin will
  use PST (IIP-0404) with Dilithium5 signatures.

- **Endpoint protocol:** BIP-78 defines an HTTPS endpoint where the
  receiver accepts the sender's unsigned transaction and returns a
  modified version with the receiver's inputs added. This model will
  be preserved but adapted for INTcoin's address format (IIP-0400)
  and transaction structure.

- **Transaction size:** Payjoin transactions in INTcoin will be
  significantly larger than Bitcoin due to Dilithium5 signature sizes
  (4,627 bytes per input). This affects fee calculations and may
  require adjusted fee-rate policies for Payjoin transactions.

- **URI scheme:** Payment URIs will use the `intcoin:` scheme with a
  `pj=` parameter for the Payjoin endpoint, replacing Bitcoin's
  `bitcoin:` URI scheme.

### Workflow (planned)

1. Receiver generates a Payjoin-capable payment URI with endpoint URL.
2. Sender constructs an initial transaction paying the receiver.
3. Sender sends the unsigned PST to the receiver's endpoint.
4. Receiver adds their own inputs and adjusts outputs.
5. Receiver returns the modified PST to the sender.
6. Sender verifies the modifications are acceptable (amount unchanged,
   fee reasonable, no new outputs to unknown addresses).
7. Sender signs and broadcasts the combined transaction.

### RPC commands (planned)

| Command | Description |
|---------|-------------|
| `payjoin_send <address> <amount> <url>` | Send payment via Payjoin |
| `payjoin_createurl <address> <amount>` | Generate receiver Payjoin URI |
| `payjoin_start` | Start Payjoin receiver endpoint |
| `payjoin_stop` | Stop Payjoin receiver endpoint |
| `payjoin_stats` | Get Payjoin usage statistics |

- IIP-0404: Partially Signed Transactions (multi-party signing)
- IIP-0400: Address Encoding (URI scheme)
- IIP-0102: Replace-By-Fee (fee bumping on Payjoin transactions)
