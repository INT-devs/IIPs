# IIP-400: Address Encoding

## Abstract

This IIP specifies the Bech32 address format for INTcoin. Addresses
encode a 20-byte public key hash using Bech32 with network-specific
human-readable prefixes (HRPs). The hash is computed as
RIPEMD160(SHA3-256(pubkey)) where pubkey is the 2,592-byte ML-DSA-87
public key.

## Motivation

A human-readable, error-detecting address format is essential for safe
fund transfer. Bech32 provides superior error detection over Base58Check
and is already widely understood in the cryptocurrency ecosystem.
Network-specific prefixes prevent cross-chain address confusion.

## Specification

### Address construction

1. Start with the 2,592-byte ML-DSA-87 public key.
2. Compute `SHA3-256(pubkey)` → 32 bytes.
3. Compute `RIPEMD160(sha3_result)` → 20 bytes.
4. Encode as Bech32 with witness version 0 and the appropriate HRP.

### Human-readable prefixes

| Network | HRP | Example |
|---------|-----|---------|
| Mainnet | `int` | `int1q...` |
| Testnet | `tint` | `tint1q...` |
| Signet | `sint` | `sint1q...` |
| Regtest | `rint` | `rint1q...` |

The `q` following the HRP separator `1` is the witness version byte
(version 0) in Bech32 encoding.

### Address length

| Component | Size |
|-----------|------|
| HRP | 3–4 characters |
| Separator | 1 character (`1`) |
| Witness version | 1 character (`q`) |
| Data (20 bytes in base-32) | 32 characters |
| Checksum | 6 characters |
| **Total** | **43–44 characters** |

### Validation

To validate an address:

1. Decode Bech32 string → HRP + data.
2. Verify HRP matches the expected network.
3. Verify witness version is 0.
4. Verify data length is 20 bytes (32 Bech32 characters).
5. Verify Bech32 checksum.

### Script construction

A valid address maps to a P2PKH script:

```
OP_DUP OP_HASH160 <20-byte-hash> OP_EQUALVERIFY OP_CHECKSIG_DILITHIUM
```

Where `<20-byte-hash>` is the RIPEMD160(SHA3-256(pubkey)) value
decoded from the address.

### Hash function choice

SHA3-256 is used instead of SHA-256 for the initial pubkey hash,
consistent with the INTcoin cryptographic stack. RIPEMD160 is
retained for the second hash to produce a 20-byte output suitable
for compact addresses.

## Rationale

Bech32 was chosen over Base58Check for:

- Better error detection (BCH code guarantees detection of up to 4
  errors in addresses up to 89 characters).
- No ambiguous characters (no 0/O/I/l confusion).
- Case-insensitive (QR code friendly).
- Standard already understood by wallets and exchanges.

The 20-byte hash (160 bits) provides a compact address while
maintaining adequate collision resistance for address uniqueness.

## Backwards Compatibility

Not applicable. This address format has been in use since genesis.
