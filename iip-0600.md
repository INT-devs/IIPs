# IIP-600: RandomX Proof-of-Work

## Abstract

This IIP specifies the use of RandomX v2.0 as the ASIC-resistant
proof-of-work algorithm for INTcoin block mining. RandomX is optimised
for general-purpose CPUs, ensuring that mining remains accessible to
commodity hardware without requiring specialised ASIC or GPU
equipment.

## Motivation

INTcoin's design goal is to keep mining decentralised and accessible.
SHA-256 (Bitcoin) and Scrypt (Litecoin) are both dominated by ASICs,
concentrating hash power in a small number of manufacturing-capable
entities. RandomX uses random program execution and memory-hard
techniques that resist efficient ASIC implementation, keeping CPU
mining competitive.

## Specification

### Algorithm

| Parameter | Value |
|-----------|-------|
| Algorithm | RandomX v2.0 |
| Mode | Full mode (2 GB dataset) |
| Flag | RANDOMX_FLAG_V2 |
| Dataset size | ~2 GB |
| Scratchpad size | 2 MB |

### Block header hashing

The RandomX input is the serialized 80-byte block header:

| Field | Size (bytes) |
|-------|-------------|
| nVersion | 4 |
| hashPrevBlock | 32 |
| hashMerkleRoot | 32 |
| nTime | 4 |
| nBits | 4 |
| nNonce | 4 |
| **Total** | **80** |

The miner increments `nNonce` (and optionally `nTime`) until the
RandomX output hash meets the difficulty target encoded in `nBits`.

### Dataset initialisation

The RandomX dataset is initialised from a key derived from the
block chain. The dataset is regenerated when the key changes (every
2,048 blocks by default). All nodes and miners must use the same
dataset for a given key to produce and verify consistent hashes.

### Difficulty encoding (nBits)

Target difficulty uses the compact `nBits` format (same as Bitcoin):

```
target = mantissa × 2^(8 × (exponent - 3))
```

Where exponent is the first byte and mantissa is the remaining three
bytes of the 4-byte nBits field.

### Proof verification

Verification uses `mining::GetPoW().CheckProof(header)`:

1. Compute the RandomX hash of the 80-byte header.
2. Interpret the 32-byte hash as a 256-bit integer (little-endian).
3. Compare against the target derived from `nBits`.
4. Valid if hash <= target.

### Mining modes

**Full mode (default):** Uses the complete ~2 GB dataset in memory.
Provides full hash rate. Required for mining.

**Light mode:** Computes dataset entries on-the-fly from the cache
(~256 MB). Slower but uses less memory. Suitable for block validation
on resource-constrained nodes.

### Thread safety

Each mining thread creates its own RandomX VM instance. The dataset
is shared (read-only) across all threads within a process. The
thread-safety fix (commit `814ff68`) ensures VMs can operate
concurrently without data races.

### Standalone miner

The standalone `intcoin-miner` binary supports two connection modes:

**RPC mode:** Connects to a running `intcoind` via JSON-RPC to fetch
block templates (`getblocktemplate`) and submit solutions
(`submitblock`).

**P2P mode:** Connects directly to the network using V2 encrypted
transport (IIP-0200), synchronizes headers, and submits blocks via
P2P messages. No `intcoind` dependency.

## Rationale

RandomX was chosen over other ASIC-resistant algorithms (Equihash,
CryptoNight, Ethash) for its proven deployment track record (Monero),
strong theoretical foundation in random program execution, and
efficient CPU utilisation that scales linearly with core count.

The 2-minute block time (IIP-0103) combined with RandomX's ~2-second
hash computation time on commodity CPUs provides a comfortable margin
for block propagation across the network before the next block is
expected.

## Backwards Compatibility

Not applicable. RandomX has been the proof-of-work algorithm since
the genesis block.
