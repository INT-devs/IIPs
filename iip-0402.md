# IIP-402: Message Signing

## Abstract

This IIP specifies the message signing format for INTcoin, adapted
from BIP-322. Because ML-DSA-87 cannot recover the public key from a
signature (unlike ECDSA), INTcoin uses self-contained signatures that
embed the full public key alongside the signature. This allows
verification without wallet access or blockchain lookups.

## Motivation

Users need to prove ownership of an address by signing an arbitrary
message. Verifiers need to check the proof without requiring access
to the signer's wallet or the blockchain. BIP-322's generic message
signing approach is adapted with the public key embedded in the
signature blob.

## Specification

### Signature format

The signature is a binary blob with the following structure:

| Field | Size (bytes) | Description |
|-------|-------------|-------------|
| Pubkey length | 2 | Big-endian uint16, always 2592 (0x0A20) |
| Public key | 2,592 | ML-DSA-87 public key |
| Signature | 4,627 | ML-DSA-87 signature |
| **Total** | **7,221** | |

The blob is Base64-encoded for text transport, producing ~9,628
characters.

**Divergence from BIP-322:** BIP-322 relies on ECDSA public key
recovery from the signature (no key embedded). INTcoin must embed
the full public key because Dilithium5 signatures do not support
key recovery.

### Message hash

The signed message is:

```
hash = SHA3-256("INTcoin Signed Message:\n" || message)
```

The magic prefix `"INTcoin Signed Message:\n"` (including the
newline) prevents signed messages from being reinterpreted as
transaction signatures.

### Signing

```
SignMessage(wallet, address, message) → base64_signature
```

1. Look up the private key for `address` in the wallet.
2. Compute `hash = SHA3-256("INTcoin Signed Message:\n" || message)`.
3. Sign `hash` with ML-DSA-87 → 4,627-byte signature.
4. Construct the blob: `[0x0A, 0x20] || pubkey || signature`.
5. Base64-encode the blob.

### Verification

```
VerifyMessage(address, message, base64_signature) → bool
```

1. Base64-decode the signature blob.
2. Read the 2-byte pubkey length (must be 2592).
3. Extract the 2,592-byte public key and 4,627-byte signature.
4. Compute `expected_hash = RIPEMD160(SHA3-256(pubkey))`.
5. Verify that `expected_hash` matches the 20-byte hash decoded
   from `address` (per IIP-0400).
6. Compute `hash = SHA3-256("INTcoin Signed Message:\n" || message)`.
7. Verify the ML-DSA-87 signature against `hash` and the embedded
   public key.
8. Return true only if both address match (step 5) and signature
   verification (step 7) succeed.

Verification requires no wallet access and no blockchain lookup.

### RPC interface

| Command | Description |
|---------|-------------|
| `signmessage <address> <message>` | Sign with wallet key |
| `verifymessage <address> <signature> <message>` | Verify (no wallet needed) |

## Rationale

Embedding the public key in the signature increases the blob size
from ~72 bytes (ECDSA) to ~7.2 KB, but eliminates the need for key
recovery — which is mathematically impossible with lattice-based
signatures. The self-contained format means any party can verify
without blockchain access, which is important for offline use cases
and cross-system integration.

The 2-byte length prefix allows future flexibility if key sizes
change in a protocol upgrade.

## Backwards Compatibility

Not applicable. This format has been in use since genesis.
