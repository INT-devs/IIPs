# IIP-415: CoinJoin Coordination Protocol

## Abstract

This IIP specifies a CoinJoin coordination protocol for INTcoin,
enabling multiple users to combine their transactions into a single
joint transaction that breaks the common-input-ownership heuristic.

## Motivation

On-chain transaction analysis can link inputs to outputs by assuming
all inputs in a transaction belong to the same entity. CoinJoin
breaks this assumption by combining inputs from multiple unrelated
users into one transaction, improving fungibility.

## Specification

### Protocol Overview

1. **Registration:** Participants register inputs and desired outputs
   with a coordinator (centralised or decentralised)
2. **Input verification:** Coordinator verifies each input exists in
   the UTXO set and meets the minimum denomination
3. **Output collection:** Each participant provides blinded output
   addresses (coordinator cannot link inputs to outputs)
4. **Transaction construction:** Coordinator builds the joint
   transaction with all inputs and outputs
5. **Signing:** Each participant signs their input(s) via PST
   (IIP-0404) round
6. **Broadcast:** Fully signed transaction is broadcast

### Denominations

Standard CoinJoin denominations for unlinkability:

| Denomination | INT |
|-------------|-----|
| Small | 10 |
| Medium | 100 |
| Large | 1,000 |
| XL | 10,000 |

### PST Integration

The joint transaction uses PST (IIP-0404) for the multi-party
signing round. Each participant receives the PST, signs their
inputs with their Dilithium5 key, and returns the partially signed
PST. The coordinator combines all partial signatures.

### Blinded Output Registration

To prevent the coordinator from linking inputs to outputs,
participants use blind signatures or a Chaumian blind scheme
during output registration.

### Equal-Output Requirement

All non-change outputs must use a standard denomination. Change
outputs are returned to the participant's wallet. The equal-output
structure is what provides the anonymity set.

## Security Considerations

- Coordinator cannot steal funds (never holds private keys)
- Coordinator cannot link inputs to outputs (blinded registration)
- Sybil attacks mitigated by requiring UTXO proof of ownership
- Denial-of-service mitigated by banning participants who fail to sign
