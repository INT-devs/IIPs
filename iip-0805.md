# IIP-805: Lightning Offers

## Abstract

This IIP specifies the Offers protocol for INTcoin Lightning,
adapting BOLT #12 for post-quantum cryptography. Offers provide
reusable payment endpoints that can generate fresh invoices on
demand, enabling recurring payments, donations, and static
payment links without invoice expiry concerns.

## Motivation

BOLT #11 invoices (IIP-0802) are single-use and expire. Offers
solve this by providing a static, reusable payment identifier that
the payer uses to fetch a fresh invoice from the recipient via the
Lightning network itself — no external server required.

## Specification

### Offer Format

Offers are encoded as Bech32m strings with prefix `into` (mainnet),
`tinto` (testnet), or `sinto` (signet).

### Offer Fields

| Tag | Description |
|-----|-------------|
| `description` | Human-readable description |
| `amount` | Optional fixed amount |
| `currency` | Optional fiat currency (for amount) |
| `quantity_max` | Maximum purchasable quantity |
| `node_id` | Recipient's Dilithium5 pubkey |
| `paths` | Blinded route hints to recipient |
| `issuer` | Optional issuer name |
| `absolute_expiry` | Optional expiry timestamp |

### Invoice Request Flow

1. Payer receives an offer (QR code, link, NFC, etc.)
2. Payer decodes the offer and constructs an `invoice_request`
3. `invoice_request` is sent via onion message (IIP-0803) to
   the offer's node, using blinded paths if provided
4. Recipient validates the request and returns a fresh `invoice`
5. Payer pays the invoice via standard Lightning payment

### Onion Messages

Offers use onion messages for the request/response flow. Onion
messages use the same Kyber1024-based routing as payment onions
(IIP-0803) but carry arbitrary data instead of HTLC instructions.

### Advantages Over IIP-0802 Invoices

| Feature | IIP-0802 Invoice | IIP-0805 Offer |
|---------|-----------------|----------------|
| Reusable | No (single-use) | Yes |
| Expiry | Yes (default 1hr) | Optional |
| Static QR | No | Yes |
| Privacy | Reveals node ID | Blinded paths |
| Recurring | No | Yes (payer-initiated) |
| Refunds | No | Built-in refund flow |

### Refund Flow

1. Payer includes a `payer_key` in the invoice request
2. Recipient can issue a refund by sending an invoice to the
   payer's key via onion message
3. Payer receives and pays the refund invoice

### Size Considerations

Offers are more compact than invoices because they omit the
payment hash and signature (the invoice fetched via onion message
contains those). A typical offer is ~4,200 characters (still large
due to the Dilithium5 node pubkey).

Offers with blinded paths instead of a direct node ID can be
shorter if the pubkey is omitted.

## Compatibility

Not compatible with Bitcoin BOLT #12 offers due to different
cryptographic primitives and address encoding.
