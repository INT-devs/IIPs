# IIP-802: Lightning Invoice Protocol

## Abstract

This IIP defines the invoice encoding format for INTcoin Lightning
payments, adapting the BOLT #11 specification with INTcoin-specific
prefixes and post-quantum signature authentication.

## Motivation

Lightning invoices encode payment details (amount, destination,
payment hash, expiry) in a compact, human-readable format. The
invoice must be signed to prove it was created by the destination
node, requiring adaptation for Dilithium5 signatures.

## Specification

### Invoice Format

```
<prefix><amount><separator><data><signature>
```

### Prefix

| Network | Prefix |
|---------|--------|
| Mainnet | `int` |
| Testnet | `tint` |
| Signet  | `sint` |

### Amount

Optional amount encoded as:
- Bare number: amount in INT (e.g., `int100` = 100 INT)
- With multiplier: `m` (milli), `u` (micro), `n` (nano)
  - `int1m` = 0.001 INT = 10 base units
  - `int1u` = 0.0001 INT = 1 base unit (minimum)

### Separator

`1` separates the human-readable part from the data part.

### Data Fields

Bech32-encoded tagged fields:

| Tag | Length | Description |
|-----|--------|-------------|
| `p` | 52 | Payment hash (SHA3-256) |
| `d` | variable | Short description (UTF-8) |
| `h` | 52 | Description hash (SHA3-256) |
| `x` | variable | Expiry time (seconds, default 3600) |
| `n` | variable | Destination node public key |
| `c` | variable | min_final_cltv_expiry (default 18) |
| `f` | variable | Fallback on-chain address |
| `r` | variable | Routing hints (for private channels) |
| `9` | variable | Feature bits |

### Signature

The invoice is signed with the destination node's Dilithium5 key.
Due to the 4627-byte signature size, Lightning invoices are
significantly longer than Bitcoin Lightning invoices (~6200+
characters versus ~300).

### Recovery ID

Bitcoin Lightning invoices include a recovery ID byte to recover
the signer's public key from the ECDSA signature. Dilithium5 does
not support key recovery, so the destination node's public key
MUST be included in the `n` tagged field (2592 bytes, Bech32-encoded).

### Example

```
int100n1pqx...  (mainnet, 100 INT)
tint1u1pqx...   (testnet, 1 base unit, "any amount")
```

### Encoding Size

| Component | Size |
|-----------|------|
| Prefix + amount | ~10 chars |
| Payment hash | ~86 chars |
| Description | variable |
| Node pubkey | ~4,150 chars |
| Signature | ~7,400 chars |
| **Total (typical)** | **~12,000 chars** |

QR codes for INTcoin Lightning invoices require high-capacity
formats (QR version 30+). Applications should support
copy-paste and NFC as alternatives to QR scanning.

## Compatibility

Not compatible with Bitcoin BOLT #11 invoices due to different
prefix, hash algorithm (SHA3-256 vs SHA-256), signature scheme,
and mandatory node pubkey field.
