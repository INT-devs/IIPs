# IIP-416: Stealth Addresses

## Abstract

This IIP specifies stealth addresses for INTcoin, enabling one-time
payment addresses derived from a published stealth meta-address.
Each payment goes to a unique address that only the recipient can
link and spend.

## Motivation

Reusing addresses degrades privacy — all payments to the same
address are trivially linked. Stealth addresses solve this by
deriving a fresh, unique address for every payment from a single
published identifier.

Unlike IIP-0405 (reusable payment codes) which requires a
notification transaction, stealth addresses embed the derivation
data in the payment transaction itself (via an OP_RETURN output),
requiring no prior interaction.

## Specification

### Stealth Meta-Address

The recipient publishes a stealth meta-address containing:
- Spending public key (Dilithium5, 2,592 bytes)
- Viewing public key (used for scanning, can be a lighter scheme)

### Payment Flow

1. Sender generates ephemeral keypair
2. Sender derives one-time address using recipient's spending pubkey
   and the ephemeral secret via SHAKE256
3. Sender includes ephemeral pubkey in an OP_RETURN output
4. Recipient scans OP_RETURN outputs using viewing key to detect
   payments
5. Recipient derives the spending key for the one-time address

### Post-Quantum Adaptation

Standard stealth addresses use ECDH for the shared secret.
INTcoin uses ML-KEM-1024 (Kyber1024) encapsulation:
- Sender encapsulates to recipient's viewing key
- Shared secret used to derive the one-time spending key
- Kyber ciphertext (1,568 bytes) stored in OP_RETURN

### Scanning

Recipients scan the blockchain for payments by:
1. Checking each OP_RETURN for a valid Kyber ciphertext
2. Decapsulating with their viewing secret key
3. Deriving the one-time address and checking if it matches an output

### View-Only Mode

The viewing key can be shared with a watch-only wallet or server to
detect incoming payments without the ability to spend them.

## Size Considerations

The OP_RETURN output containing the Kyber ciphertext adds ~1,570 bytes
per stealth payment. This is significant but acceptable for
privacy-sensitive transactions.
