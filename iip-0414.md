# IIP-414: Hardware Wallet Integration

## Abstract

This IIP specifies the integration protocol for hardware wallets
(Ledger, Trezor, and compatible devices) with INTcoin, using
Partially Signed Transactions (IIP-0404) as the interchange format
and ML-DSA-87 (Dilithium5) for on-device signing.

## Motivation

Hardware wallets provide the strongest key security by keeping
private keys on an isolated secure element. INTcoin's post-quantum
signatures (ML-DSA-87) present unique challenges:

1. **Key size:** Dilithium5 secret keys are 4,896 bytes — significantly
   larger than the 32-byte secp256k1 keys hardware wallets were
   designed for
2. **Signature size:** 4,627 bytes per signature versus 72 bytes
   for ECDSA
3. **Computation:** ML-DSA-87 signing requires more memory and
   compute than ECDSA

These constraints require firmware adaptations but the PST workflow
provides a clean abstraction that works across device types.

## Specification

### PST Workflow

Hardware wallets use the PST (IIP-0404) offline signing workflow:

```
1. Software wallet creates unsigned PST
2. PST transferred to hardware device (USB/Bluetooth/NFC)
3. Device displays transaction details for user verification
4. User confirms on device
5. Device signs with Dilithium5 key stored on secure element
6. Signed PST returned to software wallet
7. Software wallet finalises and broadcasts
```

### Key Derivation

Hardware wallets derive keys using IIP-0301 HD derivation:

```
m/44'/2210'/account'/change/index
```

The seed (typically 24 words per IIP-0301 mnemonic) is stored on
the secure element. Key derivation uses SHAKE256 (not HMAC-SHA512).

### Device Communication

#### Ledger (APDU Protocol)

| CLA | INS | Description |
|-----|-----|-------------|
| 0xE0 | 0x02 | Get public key |
| 0xE0 | 0x04 | Sign PST |
| 0xE0 | 0x06 | Get app version |
| 0xE0 | 0x08 | Get master fingerprint |

Dilithium5 public keys (2,592 bytes) and signatures (4,627 bytes)
require multi-APDU responses since the standard APDU response
buffer is 256 bytes. The Ledger app uses chained APDUs with a
continuation flag.

#### Trezor (Protobuf Protocol)

| Message | Description |
|---------|-------------|
| `INTcoinGetPublicKey` | Derive and return pubkey at path |
| `INTcoinSignPST` | Sign PST inputs with device keys |
| `INTcoinGetAddress` | Derive and display address |

PST binary is streamed to the device in chunks due to memory
constraints. The device signs one input at a time, returning
partial PSTs that the host combines.

### Secure Element Requirements

| Resource | Minimum | Notes |
|----------|---------|-------|
| RAM | 32 KB | ML-DSA-87 signing workspace |
| Flash | 128 KB | App code + Dilithium5 tables |
| Key storage | 8 KB | One Dilithium5 secret key (4,896 B) |

Current secure elements (ST33K1M5, ATECC608B) have sufficient
resources for Dilithium5. The main constraint is signing speed
(~50ms per signature on ST33K1M5).

### Display Verification

The device must display for user confirmation:
- Destination address(es) in Bech32 format (IIP-0400)
- Amount(s) in INT (4 decimal places)
- Fee amount
- Change address (marked as change)

### Multi-Signature Support

For ISMS multisig (IIP-0409), each hardware device signs its
portion of the PST. The coordinator collects partial PSTs from
all signers and combines them (IIP-0404 `combinepst`).

## Security Considerations

- Private keys never leave the secure element
- Transaction details verified on device screen (anti-malware)
- Pin/passphrase protects device access
- Firmware attestation ensures authentic Ledger/Trezor firmware
- PST format prevents signing manipulated transactions (UTXO
  data included for fee verification)
