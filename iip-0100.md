# IIP-100: OP_CHECKSIG_DILITHIUM

## Abstract

This IIP specifies OP_CHECKSIG_DILITHIUM (opcode `0xc0`), the
transaction signature verification opcode for INTcoin. It replaces
Bitcoin's OP_CHECKSIG (which uses ECDSA/secp256k1) with ML-DSA-87
(Dilithium5), a NIST-standardised post-quantum lattice-based digital
signature scheme.

## Motivation

ECDSA signatures are vulnerable to Shor's algorithm on a
sufficiently powerful quantum computer. INTcoin is designed from
genesis with post-quantum security — there is no legacy ECDSA support
and no migration path needed. OP_CHECKSIG_DILITHIUM is the only
signature verification opcode in the script engine.

## Specification

### Opcode

| Field | Value |
|-------|-------|
| Name | OP_CHECKSIG_DILITHIUM |
| Hex | `0xc0` |
| Decimal | 192 |

### Stack behaviour

Pops two items from the stack:

1. **Public key** (top) — exactly 2592 bytes (ML-DSA-87 public key).
2. **Signature** (second) — exactly 4627 bytes (ML-DSA-87 signature)
   with a 1-byte sighash type appended, making the on-stack item
   4628 bytes total.

If verification succeeds, pushes `OP_TRUE` (1). Otherwise pushes
`OP_FALSE` (0). When `SCRIPT_VERIFY_NULLFAIL` is active (default for
standard transactions), a failed verification with a non-empty
signature causes the script to fail immediately rather than pushing
false.

### Key and signature sizes

| Component | Size (bytes) |
|-----------|-------------|
| Public key | 2592 |
| Secret key | 4896 |
| Signature | 4627 |
| Sighash type suffix | 1 |
| On-stack signature item | 4628 |

Size validation is strict: any deviation from the expected sizes
causes immediate script failure.

### Sighash types

The last byte of the on-stack signature item encodes the sighash type.
The signature itself covers a SHA3-256 hash of a serialized subset of
the transaction, determined by the sighash type.

| Type | Value | Inputs signed | Outputs signed |
|------|-------|---------------|----------------|
| SIGHASH_ALL | `0x01` | All inputs (scriptCode for current) | All outputs |
| SIGHASH_NONE | `0x02` | All inputs (scriptCode for current) | None |
| SIGHASH_SINGLE | `0x03` | All inputs (scriptCode for current) | Output at same index |
| SIGHASH_ANYONECANPAY | `0x80` | Current input only | Combined with above |

SIGHASH_ANYONECANPAY may be ORed with any of the first three types.

### Sighash computation

The signature hash is computed by `SignatureHash()`:

1. Serialize the transaction with modifications per sighash type.
2. Hash the serialization with SHA3-256.
3. The resulting 32-byte digest is the message signed by ML-DSA-87.

### Verification

`TransactionSignatureChecker::CheckSignature()` performs:

1. Validate public key length == 2592 bytes.
2. Validate signature length == 4627 bytes (after stripping sighash byte).
3. Compute the sighash digest per the sighash type.
4. Call `crypto::GetSignatureScheme().Verify(pubkey, digest, signature)`.
5. Return true only if the ML-DSA-87 verification returns `CryptoError::OK`.

### Disabled opcodes

The following Bitcoin opcodes are explicitly disabled and return
`SCRIPT_ERR_DISABLED_OPCODE` if encountered:

- `OP_CHECKSIG` (0xac) — ECDSA, not supported.
- `OP_CHECKSIGVERIFY` (0xad) — ECDSA, not supported.
- `OP_CHECKMULTISIG` (0xae) — multisig not supported in the PQ model.
- `OP_CHECKMULTISIGVERIFY` (0xaf) — multisig not supported.

Multi-party signing is handled by PST (IIP-0404) workflows rather
than on-chain multisig opcodes.

### Standard script templates

**Pay-to-Public-Key-Hash (P2PKH) with Dilithium:**

```
scriptPubKey:  OP_DUP OP_HASH160 <20-byte-hash> OP_EQUALVERIFY OP_CHECKSIG_DILITHIUM
scriptSig:     <signature+sighash> <pubkey>
```

The 20-byte hash is RIPEMD160(SHA3-256(pubkey)).

## Rationale

ML-DSA-87 (NIST FIPS 204, security level 5) was chosen for its
balance of post-quantum security strength and verification
performance. At level 5 it provides 256-bit classical security and
resistance to all known quantum attacks. The larger key and signature
sizes (~7 KB per signed input vs ~100 bytes for ECDSA) are an
accepted trade-off for quantum resistance from genesis.

Multisig opcodes are disabled because Dilithium5 does not support
efficient key aggregation or threshold schemes. Multi-party
authorization is achieved through PST (Partially Signed Transaction)
workflows where each signer adds their signature independently.

## Backwards Compatibility

Not applicable. OP_CHECKSIG_DILITHIUM has been active since the
genesis block. There is no pre-quantum era to maintain compatibility
with.
