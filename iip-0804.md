# IIP-804: Lightning Gossip Protocol

## Abstract

This IIP specifies the gossip protocol for INTcoin Lightning network
graph discovery, adapting BOLT #7 with Dilithium5 signatures for
channel and node announcements.

## Motivation

Lightning nodes need to discover the network topology to route
payments. The gossip protocol propagates channel announcements,
node announcements, and channel updates across the network. All
announcements are signed to prevent spoofing.

## Specification

### Channel Announcement

Broadcast when a new public channel is confirmed on-chain.

| Field | Size | Description |
|-------|------|-------------|
| `node_signature_1` | 4,627 B | Dilithium5 signature from node 1 |
| `node_signature_2` | 4,627 B | Dilithium5 signature from node 2 |
| `features` | variable | Feature bits |
| `chain_hash` | 32 B | Genesis block hash |
| `short_channel_id` | 8 B | Funding tx block:tx:output |
| `node_id_1` | 2,592 B | Dilithium5 pubkey of node 1 |
| `node_id_2` | 2,592 B | Dilithium5 pubkey of node 2 |

**Total: ~14,500 bytes** (versus ~430 bytes in Bitcoin Lightning)

### Node Announcement

Broadcast to advertise node metadata (alias, colour, addresses).

| Field | Size | Description |
|-------|------|-------------|
| `signature` | 4,627 B | Dilithium5 signature |
| `features` | variable | Feature bits |
| `timestamp` | 4 B | Unix timestamp |
| `node_id` | 2,592 B | Dilithium5 pubkey |
| `rgb_color` | 3 B | Node colour |
| `alias` | 32 B | Node alias (UTF-8) |
| `addresses` | variable | Network addresses |

### Channel Update

Broadcast to update routing parameters (fees, timelock, capacity).

| Field | Size | Description |
|-------|------|-------------|
| `signature` | 4,627 B | Dilithium5 signature |
| `chain_hash` | 32 B | Genesis block hash |
| `short_channel_id` | 8 B | Channel identifier |
| `timestamp` | 4 B | Unix timestamp |
| `message_flags` | 1 B | Message flags |
| `channel_flags` | 1 B | Direction + disabled |
| `cltv_expiry_delta` | 2 B | Timelock delta |
| `htlc_minimum` | 8 B | Minimum HTLC amount |
| `fee_base` | 4 B | Base fee (base units) |
| `fee_proportional` | 4 B | Fee rate (millionths) |
| `htlc_maximum` | 8 B | Maximum HTLC amount |

### Gossip Queries

Nodes synchronise the network graph using query messages:

- `query_short_channel_ids`: Request specific channel announcements
- `reply_short_channel_ids_end`: End of query response
- `query_channel_range`: Request channels in a block range
- `reply_channel_range`: Channel IDs in the requested range
- `gossip_timestamp_filter`: Filter gossip by timestamp

### Bandwidth Considerations

Post-quantum signatures make gossip messages ~30x larger than
Bitcoin Lightning equivalents. Mitigations:

1. **Gossip queries:** Only request unknown channels rather than
   receiving the full graph
2. **Timestamp filtering:** Ignore old announcements
3. **Rate limiting:** Cap gossip relay bandwidth per peer
4. **Compressed gossip:** Optional zlib/zstd compression for
   bulk gossip sync (future extension)

### Validation

- Channel announcements validated against on-chain funding tx
- Signatures verified against announced node pubkeys
- Duplicate/stale announcements dropped by timestamp
- Channels with insufficient on-chain confirmations rejected
