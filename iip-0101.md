# IIP-101: Signet Block Signing

## Abstract

This IIP specifies the signet network for INTcoin, adapting BIP-325
for ML-DSA-87 post-quantum signatures. Signet is a testing network
where block production is controlled by a designated authority key
rather than open proof-of-work competition. This enables predictable
block production, controlled reorgs for testing, and a stable
environment for application development.

## Motivation

Testnet suffers from difficulty instability, spam, and unpredictable
block intervals because anyone can mine. Signet solves this by
requiring blocks to carry a valid signature from a known authority
key, while still requiring proof-of-work to test the mining path.

## Specification

### Activation

Signet is activated by the `-signet` command-line flag. It is a
separate network with its own genesis block, address prefix (`sint`),
and P2P port (22210).

### Challenge script

The signet challenge is a script embedded in the consensus parameters
that defines who may produce blocks:

```
OP_PUSHDATA2 <2592-byte ML-DSA-87 pubkey> OP_CHECKSIG_DILITHIUM
```

The authority's full 2,592-byte Dilithium5 public key is pushed
directly onto the stack. Any block whose signet commitment does not
satisfy this script is rejected.

**Divergence from BIP-325:** Bitcoin's signet uses ECDSA with a
33-byte compressed public key. INTcoin's challenge script pushes
the full 2,592-byte ML-DSA-87 public key. OP_PUSHDATA2 is required
(OP_PUSHDATA1 only supports up to 255 bytes).

### Block signing

The authority signs each block by embedding a signature in the
coinbase transaction's witness commitment before proof-of-work
grinding.

**Signing process:**

1. Build the block (header + transactions + coinbase).
2. Compute the signet block hash: `SHA3-256(block_header)` with
   nonce set to zero.
3. Sign the hash with the authority's ML-DSA-87 secret key,
   producing a 4,627-byte signature.
4. Embed the signature in the coinbase as a witness commitment.
5. Recompute the merkle root (signing modifies the coinbase).
6. Grind the nonce for proof-of-work.

**Divergence from BIP-325:** BIP-325 uses a virtual transaction pair
for signature validation. INTcoin follows the same model (IIP-56)
but with Dilithium5 signatures.

### Block validation

For signet blocks, consensus validation adds:

1. Extract the signet commitment from the coinbase.
2. Construct the virtual transaction pair per IIP-56.
3. Evaluate the challenge script against the commitment.
4. Reject the block if the script fails.

Standard proof-of-work validation (IIP-0600) also applies — signet
blocks must satisfy both the signature requirement and the
difficulty target.

### Authority key derivation

The signet authority derives its key deterministically from a seed
phrase. The `-signetkey=<hex>` flag provides the raw secret key +
public key concatenation to the daemon for block signing.

## Rationale

Signet provides a controlled testing environment where block
production rate and chain behaviour are predictable. The authority
model ensures the chain does not stall (no difficulty death spiral)
and cannot be spammed by anonymous miners.

Requiring proof-of-work in addition to the signature ensures the
mining code path is exercised during testing, unlike a pure
proof-of-authority chain.

## Backwards Compatibility

Signet is a separate network. No impact on mainnet or testnet.
