# IIP-1: IIP Purpose and Guidelines

## Abstract

An INTcoin Improvement Proposal (IIP) is a design document providing
information to the INTcoin community, describing a new feature or
change to the protocol, its processes, or its environment. This
document defines the IIP process, format requirements, and editorial
workflow.

## What is an IIP?

IIPs are the primary mechanism for proposing new features, collecting
community input on technical issues, and documenting design decisions
for INTcoin. Each IIP should provide a concise technical specification
and a rationale for the proposed change.

The IIP author is responsible for building consensus within the
community and documenting dissenting opinions.

Because INTcoin is a post-quantum cryptocurrency built on
NIST-standardised cryptographic primitives (ML-DSA-87, ML-KEM-1024,
SHA3-256), many IIPs will adapt or diverge from Bitcoin Improvement
Proposals (BIPs). When an IIP is based on an existing BIP, the
`based-on` preamble field links to the source and the body documents
all divergences. IIPs that describe features unique to INTcoin (e.g.
OP_CHECKSIG_DILITHIUM, Kyber1024 transport, RandomX proof-of-work)
have no BIP counterpart.

## IIP Types

There are three types of IIP:

**Standards Track** — proposes a change affecting the protocol, network
behaviour, block or transaction validity, or interoperability between
implementations. Standards Track IIPs require a specification section
detailed enough for independent implementation and, where applicable,
a reference implementation. Consensus-layer Standards Track IIPs
additionally require version-bits deployment parameters.

**Informational** — describes a design issue, provides background, or
offers guidance that does not require network consensus. Informational
IIPs do not propose new features and are not binding.

**Process** — describes a procedure, governance convention, or change
to the IIP workflow itself. Process IIPs require community consensus
but do not affect the protocol directly.

## IIP Layers

Each Standards Track IIP is assigned a layer indicating which part of
the system it affects. Layers correspond to reserved numbering ranges
so related proposals cluster together.

| Range | Layer | Scope |
|-------|-------|-------|
| 0001–0099 | Process | IIP workflow, editorial policy, governance |
| 0100–0199 | Consensus | Block validation, soft/hard forks, script rules |
| 0200–0299 | Peer Services | P2P protocol, transport, message formats |
| 0300–0399 | Cryptography | Post-quantum primitives, key formats, signatures |
| 0400–0499 | Applications | Wallet, address encoding, HD derivation, labels |
| 0500–0599 | API / RPC | JSON-RPC interface, REST, ZMQ notifications |
| 0600–0699 | Mining | Proof-of-work, block template, difficulty adjustment |
| 0700–0799 | Institutional | Custody, RBAC, audit, compliance, external signers |

Informational and Process IIPs use the 0001–0099 range regardless of
subject matter.

## IIP Workflow

### 1. Idea

Before writing a formal IIP, discuss the idea in the
[IIP Discussion](https://github.com/INT-devs/IIPs/discussions) forum.
The goal is to determine whether the idea has merit and is not a
duplicate before investing effort in a full specification. The author
should present the problem, proposed solution, and trade-offs.

### 2. Draft

Once discussion shows the idea has support, the author writes a formal
IIP following the format defined below and opens a pull request against
the [IIPs repository](https://github.com/INT-devs/IIPs). The pull
request title should be "IIP: \<title\>" (without a number). An editor
will assign the next available number in the appropriate layer range,
rename the file, and merge the PR with status **Draft**.

### 3. Proposed

When the author considers the IIP complete and ready for community
review, they update the status to **Proposed** via a pull request. The
IIP should have a working reference implementation (for Standards Track
IIPs) or be complete enough for community evaluation.

### 4. Final

A Standards Track IIP reaches **Final** when the change is deployed on
mainnet. For consensus-layer IIPs, this means the version-bits
deployment has activated. For other layers, this means the reference
implementation is merged and released.

Process and Informational IIPs may be marked **Active** instead of
Final when they describe ongoing procedures (like this document).

### Other statuses

- **Replaced** — a later IIP supersedes this one. The preamble notes
  the replacement.
- **Withdrawn** — the author has abandoned the proposal before it
  reached Final. May be resumed later.
- **Deferred** — the author or editors have postponed the proposal for
  reasons documented in the IIP.
- **Rejected** — community review determined the proposal should not
  be adopted. A rationale must be recorded.

## IIP Format

IIP files are Markdown (`.md`) named `iip-NNNN.md` where `NNNN` is
the zero-padded four-digit IIP number.

### Preamble

Every IIP begins with a YAML front-matter preamble enclosed in `---`
delimiters:

```yaml
---
iip: <number>
title: <title>
author: <name> <<email>>
type: <Standards Track | Informational | Process>
layer: <Consensus | Peer Services | Cryptography | Applications | API/RPC | Mining | Institutional>
status: <Draft | Proposed | Final | Active | Replaced | Withdrawn | Deferred | Rejected>
created: <YYYY-MM-DD>
requires: <IIP number(s), optional>
replaces: <IIP number(s), optional>
superseded-by: <IIP number(s), optional>
based-on: <BIP number(s) or external reference, optional>
---
```

**Field notes:**

- `author` — one or more authors, each as `Name <email>`. Multiple
  authors on separate lines.
- `layer` — omit for Process and Informational IIPs.
- `requires` — list other IIPs that must be implemented first.
- `based-on` — when the IIP adapts an existing BIP or external
  standard, cite it here. The IIP body must document all divergences
  from the source.

### Body Sections

Standards Track IIPs should include the following sections in order:

1. **Abstract** — a short (~200 word) description of the proposal.
2. **Motivation** — why the existing protocol is inadequate and what
   problem this IIP solves.
3. **Specification** — the technical specification in sufficient detail
   for an independent implementation. This is the normative section.
4. **Rationale** — why particular design decisions were made and what
   alternatives were considered.
5. **Backwards Compatibility** — how the change interacts with
   existing functionality. Consensus-layer IIPs must describe the
   activation mechanism (version bits, height, etc.).
6. **Reference Implementation** — a link to or inline listing of the
   reference implementation.
7. **References** — citations to related IIPs, BIPs, academic papers,
   or external specifications.

Informational and Process IIPs may use whatever structure is
appropriate.

### Auxiliary Files

Large diagrams, test vectors, or supplementary data may be placed in a
subdirectory named `iip-NNNN/` alongside the IIP file.

## IIP Editors

The IIP editors are responsible for:

- Assigning IIP numbers within the correct layer range.
- Reviewing PRs for format compliance (not technical merit).
- Merging well-formed IIPs and status updates.
- Maintaining the index table in README.md.

Editors do not make judgements on whether an IIP should be adopted.
Technical review and consensus are the responsibility of the
community.

Current editors:

- INTcoin Core Developers \<team@international-coin.org\>
