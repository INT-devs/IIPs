# IIP-500: ZMQ Notification Protocol

## Abstract

This IIP specifies the ZeroMQ (ZMQ) notification interface for
INTcoin, enabling real-time push notifications for blockchain events.
Applications subscribe to specific topics and receive notifications
as blocks and transactions are processed, without polling the RPC
interface.

## Motivation

Exchange integrations, payment processors, and monitoring systems
need real-time event notification. Polling the RPC interface
introduces latency, wastes bandwidth, and increases RPC server load.
ZMQ provides a publish-subscribe model where `intcoind` pushes events
to subscribers as they occur.

## Specification

### Topics

| Topic | Payload | Description |
|-------|---------|-------------|
| `hashblock` | 32-byte block hash | New block accepted to active chain |
| `hashtx` | 32-byte txid | New transaction added to mempool |
| `rawblock` | Serialised CBlock | Full block data |
| `rawtx` | Serialised CTransaction | Full transaction data |
| `sequence` | Hash + type + sequence | Ordered event sequence |

### Message Format

Each ZMQ message is a multi-part message:

```
Part 0: Topic (string, e.g. "hashblock")
Part 1: Payload (binary)
Part 2: Sequence number (u32, little-endian)
```

The sequence number increments per-topic and detects missed messages.

### Configuration

```ini
# intcoin.conf
zmqpubhashblock=tcp://127.0.0.1:28332
zmqpubhashtx=tcp://127.0.0.1:28332
zmqpubrawblock=tcp://127.0.0.1:28333
zmqpubrawtx=tcp://127.0.0.1:28333
zmqpubsequence=tcp://127.0.0.1:28332
```

Each topic can bind to a different endpoint. Multiple subscribers
can connect to the same endpoint.

### Sequence Topic

The `sequence` topic provides ordered notification of all events:

| Type Byte | Event |
|-----------|-------|
| `C` | Block connected |
| `D` | Block disconnected (reorg) |
| `A` | Transaction added to mempool |
| `R` | Transaction removed from mempool |

Payload: `<32-byte hash><1-byte type><4-byte sequence>`

This enables applications to maintain a consistent view of the
mempool and chain tip, including reorgs.

### High Water Mark

ZMQ uses a high-water mark (HWM) to control buffering. Default is
1000 messages. If a subscriber falls behind, messages are dropped
and the subscriber must resync via RPC.

### Network Binding

ZMQ endpoints should bind to `127.0.0.1` or a private interface.
ZMQ has **no authentication** — do not expose to public networks.

## Use Cases

- **Exchange deposit monitoring:** Subscribe to `hashtx` + `hashblock`
  to detect deposits and confirmations
- **Payment processing:** Subscribe to `rawtx` for real-time payment
  detection
- **Block explorer:** Subscribe to `rawblock` for live block updates
- **Monitoring dashboards:** Subscribe to `sequence` for chain state

## Security Considerations

- ZMQ provides no encryption or authentication
- Bind only to localhost or trusted networks
- Use firewall rules to restrict access
- For remote monitoring, tunnel ZMQ over SSH or VPN

## Reference Implementation

- `src/zmq/` — ZMQ notification publisher
- Currently deployed on all INTcoin fleet servers
