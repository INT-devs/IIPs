# IIP-602: Stratum V2 Mining Protocol

## Abstract

This IIP specifies the Stratum V2 mining pool protocol for INTcoin,
adapted for RandomX proof-of-work (IIP-0600) and post-quantum
encrypted connections. Stratum V2 replaces the legacy Stratum V1
JSON protocol with a binary, encrypted, and more efficient protocol
that supports job negotiation, transaction selection by miners, and
template distribution.

## Motivation

Mining pools require a standardised protocol for distributing work
to miners and receiving share submissions. Stratum V1 (JSON over
TCP) has several shortcomings:

1. **No encryption** — pool operators and ISPs can observe miner traffic
2. **No transaction selection** — pool operators choose all transactions
3. **Inefficient encoding** — JSON overhead for high-frequency messages
4. **No job negotiation** — miners cannot propose block templates

Stratum V2 addresses all of these while adding post-quantum encryption
via IIP-0200 V2 transport (Kyber1024 + ChaCha20-Poly1305).

## Specification

### Sub-Protocols

| Protocol | Purpose | Participants |
|----------|---------|-------------|
| Mining Protocol | Share submission and job distribution | Miner ↔ Pool |
| Job Negotiation | Template proposal by miners | Miner ↔ Job Negotiator |
| Template Distribution | Block template from node to pool | Node ↔ Pool |

### Connection Encryption

All Stratum V2 connections use IIP-0200 V2 transport. The ML-KEM-1024
handshake authenticates the pool to the miner using the pool's
Dilithium5 public key, preventing man-in-the-middle attacks.

### Message Framing

```
+--------+--------+--------+------------------+
| Ext.   | Msg    | Length | Payload          |
| Type   | Type   | (u24)  |                  |
| 1 byte | 1 byte | 3 byte | variable         |
+--------+--------+--------+------------------+
```

### Mining Protocol Messages

| Message | Direction | Description |
|---------|-----------|-------------|
| `SetupConnection` | Miner → Pool | Protocol version, features |
| `SetupConnection.Success` | Pool → Miner | Accepted features |
| `OpenStandardMiningChannel` | Miner → Pool | Request mining work |
| `OpenStandardMiningChannel.Success` | Pool → Miner | Channel opened |
| `NewMiningJob` | Pool → Miner | New work unit |
| `SetNewPrevHash` | Pool → Miner | New block detected |
| `SubmitSharesStandard` | Miner → Pool | Share submission |
| `SubmitShares.Success` | Pool → Miner | Share accepted |
| `SubmitShares.Error` | Pool → Miner | Share rejected |
| `SetTarget` | Pool → Miner | Difficulty adjustment |

### RandomX Adaptations

INTcoin uses RandomX PoW (IIP-0600), which differs from SHA-256d:

- **Dataset epoch:** RandomX requires a dataset derived from the
  most recent epoch block. The pool must distribute the current
  epoch seed hash so miners can initialise their RandomX dataset.
- **NewMiningJob** includes: `epoch_seed_hash` (32 bytes) and
  `epoch_number` (u32) in addition to standard fields.
- **Share validation:** Pool verifies RandomX hashes against the
  share target. Full block candidates are verified against the
  network target.

### Job Negotiation Protocol

Miners with a local `intcoind` can propose their own block templates:

1. Miner constructs template via `getblocktemplate` RPC
2. Miner sends `SetCustomMiningJob` with their template
3. Pool validates coinbase outputs (pool payout included)
4. Pool accepts or rejects the custom job

This enables censorship resistance — miners choose which
transactions to include, not the pool.

### Template Distribution Protocol

Pools connect to `intcoind` to receive block templates:

1. Pool calls `getblocktemplate` on connected node
2. Node pushes updated templates on new blocks or fee threshold
3. Pool distributes work derived from the template

### Pool Payout Methods

| Method | Description |
|--------|-------------|
| PPLNS | Pay-per-last-N-shares (variance smoothing) |
| PPS | Pay-per-share (pool absorbs variance) |
| FPPS | Full-pay-per-share (includes fee share) |
| PROP | Proportional to shares in current round |

Payout transactions use standard INTcoin addresses (IIP-0400).

### Default Ports

| Network | Stratum V2 Port |
|---------|----------------|
| Mainnet | 3333 |
| Testnet | 13333 |

## Security Considerations

- All connections encrypted via IIP-0200 (post-quantum)
- Pool authentication prevents connection hijacking
- Job negotiation prevents censorship by pool operators
- Share difficulty prevents DoS via trivial submissions
