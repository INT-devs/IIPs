# IIP-213: I2P SAM Transport

## Abstract

This IIP specifies how INTcoin nodes connect over the I2P anonymity
network using the SAM (Simple Anonymous Messaging) 3.1 protocol.
I2P provides an alternative privacy network to Tor with different
trust assumptions and network architecture.

## Motivation

Tor and I2P have different threat models. Tor uses a centralised
directory authority system; I2P uses a fully distributed network
database (NetDB). Supporting both networks provides defense in depth
— if one privacy network is compromised or censored, nodes can still
communicate through the other.

## Specification

### SAM Protocol

INTcoin connects to the local I2P router via SAM 3.1 (default
127.0.0.1:7656). The node creates a SAM session with a transient
or persistent destination, then uses STREAM CONNECT for outbound
connections and STREAM ACCEPT for inbound connections.

### I2P Address Format

I2P destinations are 32-byte binary hashes, represented as 52-character
Base32 strings with a `.b32.i2p` suffix. Serialised via ADDRv2
(IIP-0202) with network ID `0x05` (I2P).

### Session Pool

The ConnectionManager maintains a pool of pre-created SAM sessions
(up to 10) to reduce connection latency. Sessions are reused for new
connections and automatically replaced when expired or invalid.

```cpp
// Pool configuration
MAX_UNUSED_I2P_SESSIONS = 10
```

### Connection Flow

1. Node initialises SAM session: `SESSION CREATE STYLE=STREAM ID=intcoin DESTINATION=TRANSIENT`
2. For persistent identity: destination key saved to `<datadir>/i2p_private_key`
3. Inbound: `STREAM ACCEPT ID=intcoin`
4. Outbound: `STREAM CONNECT ID=intcoin DESTINATION=<base64_dest>`
5. I2P address advertised to peers via ADDRv2

### Configuration

```ini
# intcoin.conf
i2p=1                           # Enable I2P support
i2psam=127.0.0.1:7656          # SAM proxy address
i2pacceptincoming=1             # Accept inbound I2P connections
onlynet=i2p                     # Optional: I2P-only mode
```

### Timeouts

| Operation | Timeout |
|-----------|---------|
| SAM connect | 5s |
| Stream connect | 120s |
| Session create | 30s |

### Privacy Considerations

- I2P destinations are self-certifying (derived from public key)
- Persistent keys provide a stable address but link sessions
- Transient keys change on restart for better unlinkability
- All I2P transport is wrapped in IIP-0200 V2 encryption
- I2P's garlic routing provides inherent traffic analysis resistance

## Compatibility

I2P addresses require ADDRv2 (IIP-0202) for relay. I2P has higher
latency than Tor or clearnet — nodes should use generous timeouts.
