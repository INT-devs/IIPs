# IIP-301: HD Key Derivation

## Abstract

This IIP specifies hierarchical deterministic (HD) key derivation for
INTcoin wallets, adapting BIP-32 (HD wallets), BIP-39 (mnemonic
seeds), and BIP-44 (multi-account hierarchy) for the ML-DSA-87
post-quantum signature scheme. The primary divergences are the use of
SHA3-256 instead of HMAC-SHA512 for chain derivation, PBKDF2 with
HMAC-SHA3-256 for seed stretching, and the inability to derive child
public keys from parent public keys (a fundamental Dilithium5
limitation).

## Motivation

Users need deterministic wallet backup and recovery from a single
seed phrase, and wallets need structured key hierarchies for account
separation. BIP-32/39/44 provide proven models for this, but their
cryptographic primitives (HMAC-SHA512, secp256k1) must be replaced
for post-quantum security.

## Specification

### Mnemonic generation (adapted from BIP-39)

#### Wordlist

The standard BIP-39 English wordlist of 2,048 words is used without
modification.

#### Entropy to mnemonic

1. Generate random entropy of 128, 160, 192, 224, or 256 bits.
2. Compute checksum: first N bits of SHA3-256(entropy), where
   N = entropy_bits / 32.
3. Concatenate entropy || checksum.
4. Split into 11-bit groups, each indexing a word from the wordlist.

| Entropy (bits) | Checksum (bits) | Words |
|---------------|-----------------|-------|
| 128 | 4 | 12 |
| 160 | 5 | 15 |
| 192 | 6 | 18 |
| 224 | 7 | 21 |
| 256 | 8 | 24 |

**Divergence from BIP-39:** Checksum uses SHA3-256 instead of SHA-256.

#### Mnemonic to seed

```
seed = PBKDF2(
    PRF     = HMAC-SHA3-256,
    password = mnemonic_string,
    salt     = "mnemonic" || passphrase,
    c        = 2048,
    dkLen    = 64 bytes
)
```

**Divergences from BIP-39:**
- PRF is HMAC-SHA3-256 instead of HMAC-SHA512.
- HMAC block size is 32 bytes (SHA3-256 output) vs 64 bytes.
- Two PBKDF2 blocks are computed (ceil(64/32) = 2) vs one in BIP-39.

Iteration count (2,048) and output length (64 bytes) are unchanged.

### Master key derivation (adapted from BIP-32)

From the 64-byte seed:

1. Compute master entropy:
   `master_entropy = SHA3-256("INTcoin HD seed" || seed)` → 32 bytes.
2. Compute chain code:
   `chain_code = SHA3-256(master_entropy || "INTcoin HD seed")` → 32 bytes.
3. Derive master Dilithium5 keypair:
   `keypair = GenerateKeyPairFromSeed(SHA3-256("INTcoin Dilithium5 Key Generation" || master_entropy))`

The domain separator `"INTcoin HD seed"` prevents cross-chain seed
reuse from producing valid INTcoin keys.

**Divergence from BIP-32:** BIP-32 uses `HMAC-SHA512("Bitcoin seed", seed)`
and splits the 64-byte output into key material + chain code. INTcoin
uses two separate SHA3-256 calls with domain separators because
Dilithium5 key generation requires a dedicated seed-to-keypair path
(SHAKE256 entropy expansion) rather than direct interpretation of
hash output as a scalar.

### Child key derivation (adapted from BIP-32)

Given parent entropy (32 bytes), parent chain code (32 bytes), and
child index (4 bytes, big-endian):

**Hardened child** (index >= 0x80000000):

```
hash = SHA3-256(parent_chain_code || 0x00 || parent_entropy || index_be32)
child_entropy = hash[0..31]
child_chain_code = SHA3-256(child_entropy || parent_chain_code)
child_keypair = GenerateKeyPairFromSeed(
    SHA3-256("INTcoin Dilithium5 Key Generation" || child_entropy))
```

**Normal child** (index < 0x80000000):

```
hash = SHA3-256(parent_chain_code || 0x01 || parent_entropy || index_be32)
child_entropy = hash[0..31]
child_chain_code = SHA3-256(child_entropy || parent_chain_code)
child_keypair = GenerateKeyPairFromSeed(
    SHA3-256("INTcoin Dilithium5 Key Generation" || child_entropy))
```

**Divergence from BIP-32:** BIP-32 hardened derivation uses the
parent private key; normal derivation uses the parent public key
(enabling watch-only wallets to derive child public keys). INTcoin
uses parent entropy for both modes because Dilithium5 has no
algebraic relationship between parent and child public keys —
child public key derivation from a parent extended public key is
not possible.

### Derivation path (BIP-44)

```
m / 44' / 2210' / account' / change / index
```

| Level | Value | Hardened |
|-------|-------|---------|
| Purpose | 44 | Yes |
| Coin type | 2210 | Yes |
| Account | 0, 1, 2... | Yes |
| Change | 0 (external), 1 (internal) | No |
| Index | 0, 1, 2... | No |

Coin type 2210 matches the mainnet P2P port by convention.

### Extended key serialization

Extended keys are serialized in Bech32 format with SHA3-256 checksums.

**Extended public key (ixpub):**

| Field | Size (bytes) |
|-------|-------------|
| Version | 4 |
| Depth | 1 |
| Parent fingerprint | 4 |
| Child index | 4 |
| Chain code | 32 |
| Public key | 2,592 |
| **Total** | **2,637** |

Encoded with HRP `ixpub`, producing ~3,600 Bech32 characters.

**Extended private key (ixprv):**

| Field | Size (bytes) |
|-------|-------------|
| Version | 4 |
| Depth | 1 |
| Parent fingerprint | 4 |
| Child index | 4 |
| Chain code | 32 |
| Secret key | 4,896 |
| Public key | 2,592 |
| **Total** | **7,533** |

Encoded with HRP `ixprv`, producing ~10,300 Bech32 characters.

**Divergence from BIP-32:** BIP-32 uses Base58Check encoding with
`xpub`/`xprv` prefixes and 78-byte payloads. INTcoin uses Bech32
with `ixpub`/`ixprv` prefixes because (a) payloads are too large for
practical Base58, and (b) Bech32 provides better error detection.
The extended private key includes both secret and public key because
Dilithium5 cannot derive one from the other.

### Fingerprint

The parent fingerprint is the first 4 bytes of
`RIPEMD160(SHA3-256(parent_public_key))`.

### Key generation from entropy

The function `GenerateKeyPairFromSeed(seed_32)` uses a thread-local
SHAKE256 instance seeded with the 32-byte input to replace the RNG
source in liboqs key generation, producing deterministic Dilithium5
keypairs.

## Rationale

SHA3-256 replaces HMAC-SHA512 throughout the derivation chain for
consistency with the rest of the INTcoin cryptographic stack. All
hash operations in INTcoin use the SHA-3 family (SHA3-256, SHAKE256),
avoiding dependence on SHA-2 which, while not known to be quantum-
vulnerable for hashing, is not part of the post-quantum design
philosophy.

The inability to derive child public keys from parent extended public
keys means watch-only wallets cannot generate new receiving addresses
without the extended private key. This is a known limitation of all
lattice-based signature schemes and is accepted as a trade-off for
post-quantum security.

## Backwards Compatibility

Not applicable. This derivation scheme has been in use since genesis.
Wallets created with the BIP-39 mnemonic are recoverable using this
specification and the same word list.
