# IIP-115: Vault Transactions

## Abstract

This IIP specifies vault transactions for INTcoin, providing
time-delayed spending with emergency recovery using
OP_CHECKTEMPLATEVERIFY (IIP-0114) covenants.

## Motivation

Hot wallets are vulnerable to key compromise. Vaults add a time
delay to withdrawals, giving the owner a window to detect theft
and sweep funds to a recovery address using an emergency key.

## Specification

### Vault Lifecycle

```
DEPOSIT → UNVAULT (trigger) → TIMELOCK WAIT → COMPLETE
                              ↓ (during wait)
                         EMERGENCY SWEEP
```

### Components

| Key | Purpose | Storage |
|-----|---------|---------|
| Vault key | Triggers unvaulting | Hot wallet |
| Recovery key | Emergency sweep | Cold storage / hardware |
| Destination | Where funds go after timelock | Pre-committed |

### Unvault Script

```
OP_IF
  <timelock> OP_CHECKSEQUENCEVERIFY OP_DROP
  <destination_template_hash> OP_CHECKTEMPLATEVERIFY
OP_ELSE
  <recovery_pubkey> OP_CHECKSIG_DILITHIUM
OP_ENDIF
```

### Timelock Recommendations

| Use Case | Timelock | Blocks (~2min) |
|----------|----------|----------------|
| Exchange hot wallet | 24 hours | 720 |
| Personal vault | 48 hours | 1,440 |
| Institutional custody | 7 days | 5,040 |

### Monitoring

Vault owners must monitor the chain for unvault transactions. If an
unauthorised unvault is detected, the recovery key sweeps funds
before the timelock expires. Watchtower services (similar to
Lightning watchtowers) can provide this monitoring.
