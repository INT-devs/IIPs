# IIP-216: Encrypted Mempool Submission

## Abstract

This IIP specifies a protocol for submitting transactions to the
mempool in encrypted form, where the transaction content is only
revealed to miners at block construction time. This prevents
front-running and MEV extraction from the public mempool.

## Motivation

A public mempool allows observers to see pending transactions before
confirmation. This enables front-running (inserting transactions
before a known pending transaction) and MEV (Miner Extractable
Value) extraction. Encrypted submission hides transaction content
until it is included in a block.

## Specification

### Encryption Scheme

1. Sender encrypts transaction using a time-lock puzzle or
   threshold encryption scheme
2. Encrypted transaction is relayed and stored in the mempool
3. At block construction, the miner (or a threshold of miners)
   decrypts the transaction for inclusion

### Approach Options

This IIP is a placeholder for research. Two approaches are under
consideration:

**Option A: Commit-Reveal** — Sender commits hash of transaction,
then reveals after N blocks. Simple but adds latency.

**Option B: Threshold Encryption** — Transaction encrypted to a
threshold set of miners. Decryption requires cooperation of k-of-n
miners. Complex but no additional latency.

## Status

Research phase. Implementation depends on selecting an approach that
balances latency, complexity, and post-quantum security.

## Reference Implementation

Planned for Q1 2028.
