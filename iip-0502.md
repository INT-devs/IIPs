# IIP-502: Exchange Integration Protocol

## Abstract

This IIP documents the recommended integration protocol for
cryptocurrency exchanges listing INTcoin. It covers deposit
detection, withdrawal processing, address generation, and
security best practices using the INTcoin RPC interface.

## Motivation

Exchanges need a reliable, standardised approach to integrate
INTcoin. This document provides the complete workflow so exchange
engineers can implement deposits, withdrawals, and balance tracking
without ambiguity.

### Node Requirements

- Full node (`intcoind`) with `txindex=1` enabled
- Wallet loaded with HD derivation (IIP-0301)
- ZMQ notifications enabled (IIP-0500)
- Minimum 6 confirmations for deposits

### Deposit Detection

**Method 1: ZMQ + RPC (recommended)**

1. Subscribe to `hashblock` ZMQ topic (IIP-0500)
2. On new block: call `listsinceblock <last_processed_hash>`
3. Process new transactions with `confirmations >= 6`

**Method 2: Polling**

1. Poll `getblockcount` every 30 seconds
2. On new block: call `listsinceblock`

### Address Generation

```bash
# Generate unique deposit address per user
intcoin-cli getnewaddress "user_12345"
```

Addresses use Bech32 encoding (IIP-0400/0407):
- Mainnet: `int1q...`
- Testnet: `tint1q...`

### Withdrawal Processing

```bash
# Single withdrawal
intcoin-cli sendtoaddress <address> <amount>

# Batch withdrawals (lower fees)
intcoin-cli sendmany "" '{"addr1": 10.5, "addr2": 25.0}'
```

**Fee estimation:**
```bash
intcoin-cli estimatesmartfee 6  # 6-block target
```

### Hot/Cold Wallet Architecture

| Wallet | Purpose | Holds |
|--------|---------|-------|
| Hot wallet | Automated withdrawals | ~5% of reserves |
| Warm wallet | Manual top-up of hot wallet | ~15% of reserves |
| Cold wallet | Long-term storage | ~80% of reserves |

Cold wallet uses hardware signing (IIP-0414) or air-gapped PST
workflow (IIP-0404).

### Confirmation Requirements

| Amount | Confirmations | Time (~2min blocks) |
|--------|--------------|---------------------|
| < 1,000 INT | 6 | ~12 minutes |
| 1,000 - 100,000 INT | 12 | ~24 minutes |
| > 100,000 INT | 30 | ~60 minutes |

### Amount Handling

- `COIN = 10,000` (4 decimal places)
- RPC amounts are in INT (e.g., `1.5000` = 15,000 base units)
- Use string amounts in RPC to avoid floating-point precision loss
- `MAX_MONEY = 221,000,000,000,000` INT

### Monitoring

Essential RPC commands for exchange operations:

| Command | Purpose |
|---------|---------|
| `getblockchaininfo` | Chain sync status |
| `getwalletinfo` | Wallet balance and status |
| `getbalances` | Detailed balance breakdown |
| `listunspent` | UTXO management |
| `gettransaction` | Transaction details |
| `getmempoolinfo` | Mempool status |

### Error Handling

| Error | Action |
|-------|--------|
| `-5` (Invalid address) | Validate address format before sending |
| `-6` (Insufficient funds) | Top up hot wallet from warm wallet |
| `-4` (Wallet not found) | Check wallet is loaded |
| Connection refused | Node is down вҖ” alert operations team |

## Security Considerations

- RPC bound to localhost only (`rpcbind=127.0.0.1`)
- Strong RPC credentials (not default)
- Encrypted wallet (`encryptwallet`)
- Separate wallet per function (deposits, withdrawals, cold)
- Regular backups (`backupwallet`)
- Monitor for reorgs via `sequence` ZMQ topic
