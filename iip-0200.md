# IIP-200: Post-Quantum V2 Encrypted Transport

## Abstract

This IIP specifies the mandatory encrypted transport protocol for all
INTcoin peer-to-peer connections. Every connection uses ML-KEM-1024
(Kyber1024) key exchange to establish a shared secret, followed by
ChaCha20-Poly1305 authenticated encryption for all subsequent
messages. There is no unencrypted (V1) transport — post-quantum
encryption is the only supported mode.

## Motivation

P2P connections in Bitcoin are traditionally unencrypted, allowing
passive observers to monitor transaction relay, map the network graph,
and perform timing analysis. Bitcoin's BIP-324 introduced optional
encryption using X25519 key exchange, which is not quantum-resistant.

INTcoin uses ML-KEM-1024 (NIST FIPS 203) for key encapsulation,
providing post-quantum security for the transport layer to match the
post-quantum transaction signatures (IIP-0300).

## Specification

### Key exchange: ML-KEM-1024

| Parameter | Value |
|-----------|-------|
| NIST standard | FIPS 203 |
| Algorithm | ML-KEM-1024 (Kyber1024) |
| liboqs identifier | `Kyber1024` |
| Public key size | 1,568 bytes |
| Secret key size | 3,168 bytes |
| Ciphertext size | 1,568 bytes |
| Shared secret size | 32 bytes |

### Handshake protocol

The handshake uses a two-round mutual KEM encapsulation. Both parties
generate ephemeral Kyber1024 keypairs; neither key is reused across
connections.

**Phase 1 — Key exchange:**

1. Initiator generates ephemeral Kyber1024 keypair (pk_i, sk_i).
   Sends pk_i (1,568 bytes) to responder.
2. Responder generates ephemeral Kyber1024 keypair (pk_r, sk_r).
   Sends pk_r (1,568 bytes) to initiator.

**Phase 2 — KEM encapsulation:**

3. Initiator encapsulates to pk_r, producing ciphertext ct_i
   (1,568 bytes) and shared secret ss_i (32 bytes). Sends ct_i.
4. Responder encapsulates to pk_i, producing ciphertext ct_r
   (1,568 bytes) and shared secret ss_r (32 bytes). Sends ct_r.

**Phase 3 — Decapsulation:**

5. Initiator decapsulates ct_r using sk_i → ss_r.
6. Responder decapsulates ct_i using sk_r → ss_i.

Both parties now hold both ss_i and ss_r.

### Session key derivation

All keys are derived using SHA3-256 with domain-separated inputs.
`magic` is the 4-byte network magic (chain identifier).

```
session_id = SHA3-256(magic || pk_i || pk_r || ss_i || ss_r || "session_id")

initiator_send_key = SHA3-256(magic || ss_i || ss_r || "initiator_send")
responder_send_key = SHA3-256(magic || ss_i || ss_r || "responder_send")
```

The initiator uses `initiator_send_key` to encrypt and
`responder_send_key` to decrypt. The responder uses the reverse.

### Session encryption: ChaCha20-Poly1305

| Parameter | Value |
|-----------|-------|
| Algorithm | ChaCha20-Poly1305 (RFC 8439) |
| Key size | 32 bytes |
| Nonce size | 12 bytes |
| Authentication tag | 16 bytes |

The implementation uses forward-secret AEAD (FSChaCha20Poly1305):
nonces auto-increment as a counter, ensuring each message uses a
unique nonce without explicit nonce transmission.

### Message framing

Each encrypted message is framed as:

```
[3 bytes: encrypted length] [16 bytes: length auth tag]
[N bytes: encrypted payload] [16 bytes: payload auth tag]
```

| Field | Size | Notes |
|-------|------|-------|
| Encrypted length | 3 bytes | Little-endian, max ~16 MB |
| Length tag | 16 bytes | Poly1305 authentication |
| Encrypted payload | variable | The P2P message content |
| Payload tag | 16 bytes | Poly1305 authentication |
| **Overhead per message** | **35 bytes** | 3 + 16 + 16 |

Maximum message size: 4,194,304 bytes (4 MB).

### Handshake state machine

| State | Description |
|-------|-------------|
| AWAITING_KEY | Initial state; waiting for peer's public key |
| KEY_SENT | Sent our public key |
| AWAITING_CIPHERTEXT | Received peer key, sent encapsulation |
| CIPHERTEXT_SENT | Sent and received ciphertexts |
| ESTABLISHED | Session keys derived; encrypted messaging active |
| FAILED | Handshake error; connection must be dropped |

### Ephemeral key cleanup

All ephemeral Kyber1024 secret keys and shared secrets are zeroed
from memory immediately after session key derivation. Only the
derived ChaCha20-Poly1305 session keys persist for the connection
lifetime.

### Self-connection detection

The session ID is used for connection binding. If a node detects that
a connection's session ID matches one of its own recent handshakes
(via nonce matching in the VERSION message), the connection is
dropped as a self-connection.

## Rationale

ML-KEM-1024 (NIST Level 5) matches the security level of ML-DSA-87
used for transaction signatures. Using a lower KEM level would create
an asymmetry where the transport could be broken before the
signatures it protects.

Mutual KEM encapsulation (both parties encapsulate to each other)
provides forward secrecy: compromise of either party's long-term
keys does not reveal past session keys, because all KEM keypairs are
ephemeral.

ChaCha20-Poly1305 was chosen over AES-GCM for its consistent
performance across hardware without AES-NI acceleration and its
resistance to timing side channels.

V1 (unencrypted) transport is not supported. This eliminates the
downgrade-attack surface entirely — a network observer cannot force
peers into cleartext mode.

## Backwards Compatibility

Not applicable. V2 encrypted transport has been mandatory since
genesis. There is no V1 fallback.
