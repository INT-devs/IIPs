# IIP-801: Lightning Channel Management

## Abstract

This IIP specifies the payment channel state machine for INTcoin
Lightning, including channel opening, commitment transaction updates,
HTLC management, and channel closing. All signatures use ML-DSA-87
(Dilithium5) per IIP-0300.

## Motivation

Payment channels allow two parties to transact off-chain with only
the open and close transactions recorded on the blockchain. The
channel state machine must be adapted for Dilithium5's larger
signatures while maintaining the security guarantees of the
revocation-based penalty mechanism.

## Specification

### Channel Lifecycle

```
OPEN_INIT → OPEN_WAIT_FUNDING → OPEN_WAIT_CONFIRMATION →
NORMAL → (SHUTDOWN →) CLOSING → CLOSED
```

### Funding Transaction

The funding transaction is a standard on-chain transaction paying to
a 2-of-2 multisig output using `OP_CHECKSIG_DILITHIUM` (IIP-0100).
Both parties must sign commitment transactions before the funding
transaction is broadcast.

### Commitment Transactions

Each party holds a different version of the current commitment
transaction. Commitment transactions spend the funding output and
distribute the channel balance.

**Structure:**
- Input: funding outpoint (signed by both parties)
- Output 0: to-local (timelocked to `to_self_delay` blocks)
- Output 1: to-remote (immediately spendable by counterparty)
- Output 2..N: HTLC outputs (offered and received)
- Output N+1: anchor outputs (optional, for fee bumping)

**Signature size impact:** Each commitment transaction requires two
Dilithium5 signatures (9254 bytes total) versus 144 bytes for ECDSA.
This increases the minimum viable channel capacity but does not
affect the state machine logic.

### HTLC State Machine

HTLCs (Hash Time-Locked Contracts) enable multi-hop payments:

```
OFFERED → (FULFILLED | TIMED_OUT | FAILED)
```

- **Offered HTLC:** Sender commits funds locked by payment hash.
  Receiver can claim by revealing the preimage before timeout.
- **Received HTLC:** Mirror of offered HTLC from receiver's
  perspective.
- **Preimage:** 32-byte SHA3-256 preimage (not SHA-256)

### Revocation Mechanism

When a new commitment is signed, the previous commitment's revocation
secret is exchanged. Broadcasting a revoked commitment allows the
counterparty to sweep all funds using the revocation key (penalty
transaction).

Revocation keys use Dilithium5 key derivation:
```
revocation_key = SHAKE256(revocation_basepoint || per_commitment_secret)
```

### Channel Close

**Cooperative close:** Both parties sign a closing transaction
distributing the final balance. No timelocks required.

**Force close:** One party broadcasts their latest commitment
transaction. The broadcaster's output is timelocked; the
counterparty's output is immediately spendable.

**Breach remedy:** If a revoked commitment is broadcast, the
non-broadcasting party uses the revocation secret to sweep all
outputs (penalty transaction).

### Channel Reestablishment

Per BOLT #2 data loss protection, nodes exchange
`channel_reestablish` messages on reconnection containing:
- `next_commitment_number`
- `next_revocation_number`
- `your_last_per_commitment_secret`
- `my_current_per_commitment_point`

### Watchtower Integration

Watchtower clients send encrypted breach remedy data to external
watchtower servers. If a revoked commitment appears on-chain, the
watchtower broadcasts the penalty transaction on behalf of the
offline channel party.

## Size Considerations

| Component | Bitcoin | INTcoin | Notes |
|-----------|---------|---------|-------|
| Funding tx witness | ~144 B | ~9,254 B | 2x Dilithium5 sigs |
| Commitment signature | ~72 B | ~4,627 B | Per-update exchange |
| HTLC signature | ~72 B | ~4,627 B | Per-HTLC per-update |
| Revocation key | 33 B | 2,592 B | Pubkey size |

Minimum practical channel capacity should account for the larger
on-chain footprint of force-close transactions.
